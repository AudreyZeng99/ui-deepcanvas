# UI-DeepCanvas 项目操作与部署手册

本文档旨在帮助你快速启动本地预览，并掌握 GitHub Pages 和 Vercel 的持续部署流程。

## 0. 版本管理建议（试验性开发 / 团队预览）

当你需要做试验性开发（更新 pages，让团队先看效果，之后再决定是否发布）时，建议采用下面的分支策略：

- `master`：稳定版本（对外展示 / 线上生产）。只通过 PR 合并进入，建议加保护规则。
- `develop`：集成分支（阶段性可用版本）。用于多功能汇总、联调。
- `exp-*` 或 `feat-*`：试验/功能分支（从 `develop` 切出）。团队评审完成后合并回 `develop`，需要发布时再由 `develop` 合并到 `master`。

推荐流程（给团队看但不影响正式版本）：

```bash
git checkout develop
git pull
git checkout -b exp-personal-workspace
git push -u origin exp-personal-workspace
```

然后在 GitHub 上发起 PR（目标分支选 `develop`），让团队成员在 PR 上 review，并通过部署平台的 Preview URL 直接体验。

## 1. 本地启动与预览

在终端中执行以下命令即可启动开发环境：

```bash
npm run dev
```

*   **说明**：该命令会启动本地开发服务器。
*   **访问地址**：通常为 `http://localhost:5173`（请以终端实际输出为准）。
*   **特点**：支持热更新（Hot Reload），修改代码后浏览器会自动刷新。

如果你想预览**打包后**的生产环境效果（更接近线上真实体验）：

```bash
npm run build
npm run preview
```

## 2. GitHub Pages 持续部署

本项目已配置自动化工作流（GitHub Actions），因此**无需手动运行构建命令**。

### 核心操作：
当前工作流会在推送到以下分支时触发部署：

- `master`：更新正式线上地址
- `exp-*`：更新分支预览地址（不会覆盖 master）

你可以同时维护多个 `exp-*` 分支给团队看，每个分支都有独立的 Pages 预览地址：

`https://AudreyZeng99.github.io/ui-deepcanvas/previews/<分支名>/`

建议：

- 线上正式版本：只用 `master` 触发部署
- 试验性预览：推送到 `exp-*` 分支（Pages 预览）或使用 Vercel Preview（见下方）

**在终端执行以下三步曲：**

1.  **添加修改**：
    ```bash
    git add .
    ```
2.  **提交修改**：
    ```bash
    git commit -m "描述你的修改内容"
    ```
3.  **推送到远程仓库**：
    ```bash
    git push origin master
    ```

全部复制
```bash
git add .
git commit -m "feat: add Admin Material Management page"
git push origin master
```

### 首次设置（必读）：
如果你是第一次部署，请确保在 GitHub 仓库中完成了以下设置：
1.  进入仓库页面，点击顶部 **Settings** (设置)。
2.  在左侧侧边栏找到 **Pages**。
3.  在 **Build and deployment** 下的 **Source** 选项中，选择 **GitHub Actions**。
    *   *注意：不要选 "Deploy from a branch"，因为我们使用的是自定义的 Action 脚本。*

设置完成后，每次 `git push` 后，你可以点击仓库顶部的 **Actions** 标签页查看部署进度。

## 3. Vercel 持续部署

Vercel 的部署最为智能和自动化，推荐使用其 GitHub 集成功能。

### 方式一：自动化部署（推荐）
如果你已经在 Vercel 控制台导入了此 GitHub 项目：
*   **推荐绑定**：将 Vercel 的 Production Branch 设为 `master`。
*   **Preview 机制**：对 `exp-*`、`feat-*` 或 PR 分支，Vercel 会自动生成预览地址（Preview URL），非常适合试验性开发给团队提前体验。
*   **原理**：Vercel 监听 GitHub 仓库变动，生产分支更新会更新线上版本，其他分支/PR 会生成独立的预览版本。

### 方式二：手动命令行部署（备用）
如果你不想通过 GitHub 触发，或者想在本地直接部署到 Vercel：

1.  **安装依赖**（如果未安装）：
    ```bash
    npm i -g vercel
    ```
2.  **执行部署**：
    ```bash
    npx vercel --prod
    ```
    *   按提示操作即可将本地当前代码直接发布到线上。

## 总结

- 试验性开发给团队看：推送到 `exp-*` 或 `feat-*` 分支，给团队发 Pages 预览地址或 Vercel Preview URL
- 决定发布：将试验分支合并到 `develop`，最终由 `develop` 合并到 `master`
- 注意：本仓库远程已存在一个名为 `exp` 的分支，因此不能再创建 `exp/xxx` 这种“带斜杠”的分支名；请使用 `exp-xxx` 形式

日常最常用的提交推送命令：

```bash
git add . && git commit -m "更新内容" && git push
```

这两个平台都会自动为你完成剩下的构建和发布工作。
